## Theory (SE-Level)

### What Is Cloudflare's Web3 Gateway?

Cloudflare's Web3 gateway bridges two worlds: the decentralized web (IPFS, Ethereum) and the familiar HTTP web. It lets you serve content stored on IPFS or resolved via ENS (Ethereum Name Service) at a custom domain like `macksportreport.com/nft-gallery` — without requiring your users to install any special software or browser extensions.

**Two things it does:**
1. **IPFS Gateway:** Fetches content from the IPFS peer-to-peer network and serves it over standard HTTP(S) at your domain
2. **ENS Resolution:** Resolves `.eth` domain names to their associated IPFS content and serves that content at your custom domain

### IPFS: InterPlanetary File System

IPFS is a peer-to-peer distributed file system. Instead of addressing content by location (a URL pointing to a specific server), IPFS addresses content by its cryptographic hash — a **Content Identifier (CID)**.

**How CIDs work:**
```
Traditional URL: https://server.com/path/to/file.html
Location-addressed: content lives at this server, at this path

IPFS CID: bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi
Content-addressed: content is identified by what it IS, not where it IS
```

The CID is the SHA-256 hash of the content (plus metadata). If you change even one byte of the file, the CID changes completely. This means:
- **Immutable content:** once published, a CID always refers to the same content
- **Verifiable:** anyone can confirm the content matches its CID
- **Distributed:** the content can be hosted by any IPFS node, and the CID still works

**IPFS network operation:**
1. You "pin" a file to IPFS (either on your own node or via a pinning service like Pinata or Web3.Storage)
2. The file is distributed to IPFS nodes that "pin" it
3. Clients retrieve the file by asking the IPFS network: "who has CID `bafybe...`?"
4. IPFS routes them to a node that has it

**The availability problem:** Content on IPFS is only available as long as at least one node is "pinning" it. If no node pins a file, it becomes unreachable. Cloudflare is a gateway to IPFS — it connects to the IPFS network and retrieves content on behalf of HTTP clients. But it doesn't pin content itself. Your content must be pinned elsewhere.

### ENS: Ethereum Name Service

ENS is a naming system built on the Ethereum blockchain. It maps human-readable names like `macksportreport.eth` to machine-readable identifiers like:
- Ethereum wallet addresses
- IPFS content hashes
- URLs

**ENS TLD is `.eth`** — these are not traditional DNS domains. They live on the Ethereum blockchain, not in ICANN's root DNS.

**How ENS resolves to IPFS:**
1. ENS name `macksportreport.eth` has a `content` record pointing to an IPFS CID
2. Cloudflare's gateway resolves the ENS name via Ethereum (using a node or provider)
3. Retrieves the IPFS content at the resolved CID
4. Serves it over HTTP at a URL you control

**The `.eth.limo` gateway:**
Since browsers can't natively resolve `.eth` names, services like `eth.limo` provide HTTP gateways. `macksportreport.eth.limo` resolves the ENS name and serves the content. Cloudflare provides similar functionality through its own gateway.

### DNSLink

DNSLink is a convention that lets you associate a traditional DNS domain with IPFS content using a TXT record:

```
_dnslink.macksportreport.com TXT "dnslink=/ipfs/bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi"
```

IPFS gateways (including Cloudflare's) check for DNSLink records when you request content via a hostname. If a DNSLink TXT record exists, the gateway resolves it to the IPFS CID and serves that content.

This lets you "point" a traditional domain at IPFS-hosted content without changing your MX records, maintaining your regular DNS while also being accessible via IPFS.

### Setting Up a Web3 Hostname in Cloudflare

Cloudflare's Web3 hostname feature in the dashboard lets you:
1. Create a hostname on your zone (e.g., `ipfs.macksportreport.com`)
2. Point it to an IPFS CID or an ENS name
3. Cloudflare serves the IPFS content at that hostname over HTTPS

The user visits `https://ipfs.macksportreport.com` → Cloudflare fetches from IPFS → serves with your SSL certificate.

## Deep Dive (Architect-Level)

### Cloudflare's IPFS Gateway Architecture

```
User browser
    │
    │ HTTPS request
    ▼
Cloudflare Edge PoP
    │
    │ CID lookup in IPFS DHT (via Cloudflare's IPFS nodes)
    ▼
IPFS Network ──────► Pinning Service (Pinata/Web3.Storage/NFT.Storage)
                     (source of your content)
    │
    │ Content retrieved
    ▼
Cloudflare Edge (caches content by CID)
    │
    │ HTTP response
    ▼
User browser
```

**Important:** Since IPFS content is immutable (CID = content hash), Cloudflare can cache it indefinitely with very high cache hit ratios. A CID served once can be cached at every edge PoP globally. This makes IPFS + Cloudflare gateway an excellent combination for static, immutable content.

### CID Versions: CIDv0 vs CIDv1

**CIDv0:** Starts with `Qm...` (base58-encoded SHA-256 hash)
```
Example: QmXFGqtSMn3tKThCoNkiN6KoJmJQ7D1GQegzLQDoFPLBt4
```
Legacy format, shorter, widely supported.

**CIDv1:** Starts with `bafy...` or `bafk...` (multibase-encoded)
```
Example: bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi
```
Preferred format, supports multiple hash functions, safe to use in subdomains (lowercase).

**Subdomain gateway pattern (CIDv1 required):**
```
https://bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi.ipfs.cloudflare-ipfs.com
```
Each CID gets its own origin for origin isolation (security). Requires CIDv1 because CIDv0 is uppercase-incompatible with DNS.

**Path gateway pattern (CIDv0 or v1):**
```
https://cloudflare-ipfs.com/ipfs/QmXFGqtSMn3tKThCoNkiN6KoJmJQ7D1GQegzLQDoFPLBt4
https://cloudflare-ipfs.com/ipfs/bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi
```

### NFT Metadata on IPFS

One of the most practical use cases: NFT metadata stored on IPFS.

```json
// NFT metadata JSON (stored on IPFS)
{
  "name": "Mack's Sport Report #001",
  "description": "Founding member NFT for Mack's Sport Report premium community",
  "image": "ipfs://bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi/nft-001.png",
  "attributes": [
    {"trait_type": "Access Level", "value": "Premium"},
    {"trait_type": "Edition", "value": "Founding"}
  ]
}
```

The NFT smart contract stores the IPFS CID on-chain as a `tokenURI`. Marketplaces (OpenSea, Blur, etc.) fetch the metadata from IPFS via a gateway. Cloudflare's gateway (`cloudflare-ipfs.com`) is one of the most reliable and fast gateways they use.

### ENS Resolution Flow in Detail

```
1. User requests: GET https://macksportreport.eth.limo/
2. eth.limo resolves ENS name via Ethereum JSON-RPC
3. ENS contract returns content record: ipfs://bafybe...
4. eth.limo fetches from IPFS (via its own IPFS nodes or Cloudflare gateway)
5. Serves to user with headers indicating IPFS source

Or with Cloudflare Web3 hostname:
1. User requests: GET https://web3.macksportreport.com/
2. Cloudflare Web3 config: this hostname → ENS name "macksportreport.eth"
3. Cloudflare resolves ENS → gets IPFS CID
4. Cloudflare fetches from IPFS
5. Serves to user with valid SSL for macksportreport.com
```

### Web3 Gateway Limitations to Know

| Limitation | Detail |
|-----------|--------|
| Content availability | IPFS content only available if at least one node pins it |
| No Cloudflare pinning | Cloudflare does NOT pin your content |
| Ethereum node dependency | ENS resolution requires Ethereum node access |
| Mutability paradox | Content is immutable — updates require new CID, new config |
| Latency | IPFS retrieval can be slow for unpopular/new content |
| Not for dynamic content | Only works for static/pre-built content |

### When to Actually Recommend This

**Recommend for:**
- NFT projects needing metadata served reliably at scale
- Decentralized websites / Web3 projects
- Immutable content archives (research, legal records)
- Customers already using IPFS who want Cloudflare's CDN in front of it

**Don't recommend for:**
- Standard websites (overengineered for the use case)
- Dynamic content (IPFS is static/immutable)
- Customers without existing Web3 infrastructure
- Situations where "we don't want to deal with IPFS pinning"

**SE Honesty:** For most Cloudflare customers, this feature is irrelevant. Know it exists. Know what it does. Know when it applies. Don't lead with it unless the customer has a clear Web3 use case.

## Hands-On Lab

### Lab 1: Interact with Cloudflare's Public IPFS Gateway

```bash
# Cloudflare's public IPFS gateway (no account required)
# Base URL: https://cloudflare-ipfs.com

# Fetch a file by CID (path gateway)
curl -s "https://cloudflare-ipfs.com/ipfs/bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi"   -H "User-Agent: curl/8.0" | head -c 500

# Check response headers
curl -s -I "https://cloudflare-ipfs.com/ipfs/QmXFGqtSMn3tKThCoNkiN6KoJmJQ7D1GQegzLQDoFPLBt4"

# Look for:
# X-IPFS-Gateway-Version
# Cache-Control: public, max-age=29030400  (immutable content, cached forever)
# Content-Type: (determined from file)
```

### Lab 2: Upload Content to IPFS via Pinata

```bash
# Prerequisites: Pinata account (free tier: 1GB), API key

PINATA_API_KEY="your-pinata-api-key"
PINATA_SECRET="your-pinata-secret"

# Create a simple HTML file for IPFS
cat > index.html << 'HTML'
<!DOCTYPE html>
<html>
<head><title>Mack's Sport Report on IPFS</title></head>
<body>
  <h1>Mack's Sport Report</h1>
  <p>This content is stored on IPFS and served via Cloudflare's gateway.</p>
  <p>CID (Content Identifier) is the hash of this file's content.</p>
</body>
</html>
HTML

# Upload to IPFS via Pinata
IPFS_RESULT=$(curl -s -X POST "https://api.pinata.cloud/pinning/pinFileToIPFS"   -H "pinata_api_key: ${PINATA_API_KEY}"   -H "pinata_secret_api_key: ${PINATA_SECRET}"   -F "file=@index.html"   -F 'pinataMetadata={"name": "macksportreport-homepage"}')

echo $IPFS_RESULT | python3 -m json.tool

# Extract CID
CID=$(echo $IPFS_RESULT | python3 -c "import sys, json; print(json.load(sys.stdin)['IpfsHash'])")
echo "Content CID: $CID"
echo "Access via: https://cloudflare-ipfs.com/ipfs/$CID"

# Verify content is accessible
curl -s "https://cloudflare-ipfs.com/ipfs/$CID"
```

### Lab 3: Create a Web3 Hostname via API

```bash
# Create a Web3 hostname on your zone
CID="bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi"

curl -s -X POST "https://api.cloudflare.com/client/v4/zones/${ZONE_ID}/web3/hostnames"   -H "Authorization: Bearer ${CF_API_TOKEN}"   -H "Content-Type: application/json"   --data "{
    "name": "ipfs.macksportreport.com",
    "description": "IPFS content gateway for macksportreport.com",
    "target": "ipfs",
    "dnslink": "${CID}"
  }" | python3 -m json.tool

# List all Web3 hostnames
curl -s "https://api.cloudflare.com/client/v4/zones/${ZONE_ID}/web3/hostnames"   -H "Authorization: Bearer ${CF_API_TOKEN}" | python3 -m json.tool
```

### Lab 4: Configure DNSLink Record

```bash
# Add _dnslink TXT record pointing to IPFS CID
CID="bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi"

curl -s -X POST "https://api.cloudflare.com/client/v4/zones/${ZONE_ID}/dns_records"   -H "Authorization: Bearer ${CF_API_TOKEN}"   -H "Content-Type: application/json"   --data "{
    "type": "TXT",
    "name": "_dnslink",
    "content": "dnslink=/ipfs/${CID}",
    "ttl": 300
  }" | python3 -m json.tool

# Verify the DNSLink record
dig TXT _dnslink.macksportreport.com +short
```

### Lab 5: Resolve an ENS Name

```bash
# Query ENS resolution via Cloudflare's Ethereum gateway
# Cloudflare provides an Ethereum JSON-RPC endpoint

# Look up an ENS name's content record (requires ABI encoding, simplified here)
curl -s -X POST "https://cloudflare-eth.com"   -H "Content-Type: application/json"   --data '{
    "jsonrpc": "2.0",
    "method": "eth_call",
    "params": [
      {
        "to": "0x4976fb03C32e5B8cfe2b6cCB31c09Ba78EBaBa41",
        "data": "0x3b3b57de"
      },
      "latest"
    ],
    "id": 1
  }'

# Or use a simple ENS resolution service
curl -s "https://api.ensideas.com/ens/resolve/macksportreport.eth" | python3 -m json.tool

# Or check via eth.limo gateway
curl -s -I "https://macksportreport.eth.limo" 2>&1 | head -20
```

## Competitive Context

| Gateway | Provider | IPFS | ENS | Custom Domain | Pricing |
|---------|----------|------|-----|--------------|---------|
| cloudflare-ipfs.com | Cloudflare | Yes | Via web3 hostname | Yes | Free (gateway) |
| gateway.pinata.cloud | Pinata | Yes | No | Yes (paid) | Paid tiers |
| ipfs.io | Protocol Labs | Yes | No | No | Free |
| dweb.link | Protocol Labs | Yes | No | No | Free |
| eth.limo | Community | Yes (via ENS) | Yes | No | Free |
| infura.io/ipfs | ConsenSys | Yes | Via API | Yes | Paid |

**Cloudflare's advantages:**
- Fastest IPFS gateway due to global CDN (content cached at all PoPs once fetched)
- SSL/TLS with your own certificate via custom hostname
- Already in your Cloudflare dashboard
- No IPFS node management
- Immutable content cached indefinitely (CIDs never change)

**When to use competitors:**
- Pinata: if you need reliable pinning, not just gateway (Cloudflare doesn't pin)
- Protocol Labs gateways: if you want free access without Cloudflare dependency
- Infura: if you need Ethereum + IPFS in one managed API platform
