## Theory (SE-Level)

### The Two TLS Connections

When Cloudflare is in front of your site, there are actually **two separate TLS connections**:

```
User Browser ←──TLS #1──→ Cloudflare Edge ←──TLS #2──→ Your Origin Server
```

- **TLS #1 (Browser to Cloudflare):** Cloudflare terminates this. Uses your edge certificate.
- **TLS #2 (Cloudflare to Origin):** Controlled by your SSL/TLS mode setting. Can be encrypted or not.

The SSL/TLS > Overview setting controls TLS #2. TLS #1 is always encrypted (Cloudflare always uses HTTPS with users).

### The Four Encryption Modes

**OFF**
```
Browser ──HTTP──→ Cloudflare ──HTTP──→ Origin
```
- All traffic is unencrypted between users and origin
- Almost never appropriate
- Only use: internal testing with no sensitive data
- What users see: `http://` in browser

**FLEXIBLE**
```
Browser ──HTTPS──→ Cloudflare ──HTTP──→ Origin
```
- User to Cloudflare: encrypted ✓
- Cloudflare to origin: NOT encrypted ✗
- Users see the padlock (HTTPS), but it's a false sense of security
- Use case: Origin that doesn't have an SSL cert (legacy systems)
- Risk: Traffic between Cloudflare and origin is plaintext on the network
- Cloudflare's recommendation: Avoid if at all possible

**FULL**
```
Browser ──HTTPS──→ Cloudflare ──HTTPS──→ Origin
```
- Both connections encrypted ✓
- Origin has a certificate, but Cloudflare does NOT verify it
- Cloudflare accepts any cert (expired, self-signed, wrong hostname)
- Use case: Origin has a certificate but it's self-signed or expired
- Risk: Vulnerable to MITM between Cloudflare and your origin if attacker can intercept that leg
- Better than Flexible, but not ideal

**FULL (STRICT)**
```
Browser ──HTTPS──→ Cloudflare ──HTTPS──→ Origin (cert validated ✓)
```
- Both connections encrypted ✓
- Cloudflare validates the origin certificate: must be valid, not expired, correct hostname
- Use Cloudflare's Origin CA cert or any trusted CA cert on your origin
- This is the **recommended setting for all production sites**
- Risk: If your cert expires on the origin, your site goes down

### Additional Settings on the Overview Page

**Always Use HTTPS:**
- Automatically redirects `http://` requests to `https://`
- Does a 301 or 302 redirect
- Alternative: Set up a Redirect Rule (more flexible)

**HSTS (HTTP Strict Transport Security):**
- Tells browsers to always use HTTPS for your domain
- Once set, users can't bypass HTTPS warnings for your domain
- `includeSubDomains`: applies to all subdomains
- `preload`: adds your domain to the browser HSTS preload list (can't be undone easily)
- **Warning:** Enable HSTS only when you're committed to HTTPS forever

**Minimum TLS Version:**
- TLS 1.0: Legacy, vulnerable (POODLE, etc.) — avoid
- TLS 1.1: Deprecated
- TLS 1.2: Safe minimum for most use cases
- TLS 1.3: Best performance (0-RTT handshake) and security — default on CF
- Recommendation: Set minimum to TLS 1.2, accept TLS 1.3

**Opportunistic Encryption:**
- For HTTP-only sites, serves encryption hints so browsers can upgrade
- Useful transitional feature if you can't force HTTPS yet

**TLS 1.3:**
- Toggle to enable/disable TLS 1.3
- Should always be enabled
- Enables 0-RTT (zero round trip time) resumption for returning users

## Deep Dive (Architect-Level)

### Why FULL (STRICT) Is the Only Right Answer

The attack vector for FULL (non-strict):
1. Attacker gets between Cloudflare's edge PoP and your origin (feasible for on-premise origins on shared hosting, ISP-level interception)
2. Presents a self-signed cert to Cloudflare
3. Cloudflare accepts it (FULL mode doesn't validate)
4. Attacker is now doing full MITM on your origin traffic

FULL STRICT prevents this by requiring a valid cert. Options for your origin:
- **Cloudflare Origin CA certificate** (Module 2.4) — free, generated by Cloudflare, trusted by Cloudflare only
- **Let's Encrypt certificate** — free, trusted publicly, auto-renews
- **Commercial CA certificate** — paid, longer validity

### TLS 1.3 and 0-RTT

TLS 1.3 introduced 0-RTT (zero round trip time) session resumption:
- First connection: normal 1-RTT handshake
- Subsequent connections: client can send data in the very first packet using session resumption keys
- This eliminates ~100ms of handshake latency on repeat visits

**Security note:** 0-RTT data is vulnerable to replay attacks (attacker replays the first packet). Safe for GET requests (idempotent), but use with care for POST requests that mutate state. Cloudflare handles this automatically for most use cases.

### The HSTS Preload Risk

HSTS preload is serious:
1. You add your domain to Chrome's/Firefox's hardcoded HSTS preload list
2. Browsers refuse to connect via HTTP to your domain even if you change the HSTS header
3. Removal from the preload list takes months (browser version release cycle)
4. If you ever need HTTP (certificate expired, need to debug, switching providers), you can't

**Best practice:** Enable HSTS with `max-age=31536000` (1 year) first. Wait 1 year. Only then add `preload` if you're absolutely committed.

### TLS Fingerprinting

JA3/JA4 fingerprinting analyzes TLS handshake parameters to identify clients. Cloudflare exposes JA3 fingerprints via:
- `cf.tls_client_hello.ja3_hash` in Ruleset Engine
- `cf-bot-management` header

This is used by bot management to identify automated clients even when they spoof user-agent headers — the TLS handshake characteristics of automated tools differ from real browsers.

### Certificate Transparency (CT Logs)

All publicly trusted certificates must be logged in Certificate Transparency logs. Cloudflare automatically submits edge certificates. This means:
- Anyone can see that `macksportreport.com` has a certificate issued by X CA
- Attackers can discover your subdomains via CT log monitoring
- You should monitor CT logs for unauthorized certificate issuance (use CAA records to prevent it)

- Currently selected mode is highlighted

**Configuration summary panel (right side):**
Shows currently configured settings:
- SSL mode
- HSTS status
- Min TLS version
- Always Use HTTPS
- Certificate type

**Quick Actions:**
- Link to Edge Certificates
- Link to Origin Server settings

**Recommendation engine:**
- If you're on Flexible or Off, Cloudflare shows a warning banner recommending upgrade

### Changing the SSL Mode

1. Click the desired mode tile (Full Strict recommended)
2. Confirm the change
3. Change takes effect within seconds globally

**What can break when you change from Flexible to Full/Strict:**
- Origin must already have a valid HTTPS certificate
- If it doesn't, requests to origin fail and users see errors
- Always test after changing modes

## Hands-On Lab

### Lab 2.1: Verify and Configure SSL/TLS Mode

**Step 1: Check current mode**
```
SSL/TLS → Overview
```
What mode are you currently on? ________________

**Step 2: Verify your current TLS configuration from the browser**
```bash
# Check TLS version and cipher suite being used
curl -I --tlsv1.2 https://macksportreport.com
# Note: SSL version, cipher in -v output

# Detailed TLS handshake info
openssl s_client -connect macksportreport.com:443 -brief
# Shows: Protocol: TLSv1.3, Cipher: TLS_AES_256_GCM_SHA384

# Or use online tool: https://www.ssllabs.com/ssltest/
```

**Step 3: Set minimum TLS version to 1.2**
```
SSL/TLS → Overview → Minimum TLS Version → TLS 1.2
```

**Step 4: Enable "Always Use HTTPS"**
```
SSL/TLS → Overview → Always Use HTTPS → On
```

Verify:
```bash
curl -I http://macksportreport.com
# Should return: HTTP/1.1 301 Moved Permanently
# Location: https://macksportreport.com/
```

**Step 5: Enable TLS 1.3**
```
SSL/TLS → Overview → TLS 1.3 → On
```

Verify:
```bash
openssl s_client -connect macksportreport.com:443 -tls1_3 -brief
# Should show TLSv1.3
```

**Step 6: Test if your origin supports HTTPS for FULL STRICT mode**
```bash
# Connect directly to your origin (replace with actual origin IP)
# This tests origin cert, not Cloudflare's edge cert
curl -I https://your-origin-ip \
  -H "Host: macksportreport.com" \
  --resolve "macksportreport.com:443:your-origin-ip"
# If you get a cert error, origin doesn't have a valid cert → use Full (not Strict) for now
# If 200/301, origin cert is valid → enable Full Strict
```

**Step 7: Run SSL Labs test**
```bash
open https://www.ssllabs.com/ssltest/analyze.html?d=macksportreport.com
```

## Demo Script (2 Minutes)

> Use when explaining SSL/TLS to a customer confused about the difference between modes

"There's a common misconception about Cloudflare and HTTPS. When you're behind Cloudflare, there are actually two encrypted connections — one from the user to Cloudflare's edge, and one from Cloudflare's edge back to your origin server. The SSL mode setting controls that second connection.

On 'Flexible' mode, the user sees the padlock, but the traffic between Cloudflare and your server is unencrypted. That's not secure — it just hides the insecurity from the user's view. We recommend 'Full Strict' — both connections are encrypted, and we validate your origin's certificate.

The way to get there without breaking things is to use our Origin CA certificate — it's free, we generate it, and it's trusted by our edge. You install it on your origin, flip the mode to Full Strict, and now you have end-to-end encryption that we actually validate. Your SSL Labs grade will typically jump from B or C to A."
