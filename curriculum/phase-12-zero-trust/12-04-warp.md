## Theory (SE-Level)

WARP is Cloudflare's device client — a software agent installed on laptops, phones, and tablets that connects the device to Cloudflare's network. Think of it as the "last mile" of Zero Trust: connecting the endpoint to the policies you've configured in Gateway and Access.

**Two distinct products share the WARP brand:**

1. **Consumer WARP (1.1.1.1 + WARP):** Free app for individuals. Routes traffic through Cloudflare for privacy and performance. Not configurable by enterprises.

2. **Zero Trust WARP:** Enterprise-managed client. Connected to your Zero Trust account. Enforces Gateway DNS/HTTP policies, enables device posture checks, provides private network access via Tunnel, and routes through your organizational policies.

This module focuses exclusively on Zero Trust WARP.

**What Zero Trust WARP does when installed and enrolled:**
- DNS queries route through Cloudflare Gateway (Gateway DNS policies apply)
- HTTP/HTTPS traffic routes through Cloudflare Gateway (HTTP policies apply, if configured)
- Device posture checks run (is the device compliant?)
- Private networks accessible via Cloudflare Tunnel routes
- All traffic encrypted via WireGuard to the nearest Cloudflare PoP

**WARP is not a traditional VPN.** There's no "connect/disconnect" in most configurations. It's always-on, managed by IT, and the user experience is minimal — an icon in the system tray showing status.

## Deep Dive (Architect-Level)

### WireGuard Protocol

WARP uses WireGuard as the transport layer, not OpenVPN or IPSec. WireGuard advantages:
- Smaller codebase (fewer attack surface)
- Better performance (kernel-level, less overhead)
- Faster handshakes
- Roaming: handles IP address changes better than OpenVPN (important for mobile)

The WireGuard tunnel goes from the device to the nearest Cloudflare PoP. Traffic exits Cloudflare to the internet (or stays inside Cloudflare's network for private resources).

### WARP Modes

| Mode | DNS Through CF | HTTP Through CF | Private Networks | Use Case |
|---|---|---|---|---|
| **Gateway with WARP** | Yes | Yes | Yes (via Tunnel routes) | Full Zero Trust deployment |
| **Gateway with DoH** | Yes | No | No | DNS-only filtering, BYOD |
| **WARP to Tunnel** | No | No | Yes | Private network access only |
| **Proxy** | No | Yes (HTTP proxy only) | No | Browser-only filtering |

**Most common enterprise deployment:** Gateway with WARP — full traffic inspection and private network access.

### Device Posture Checks

WARP collects device state and can enforce posture requirements as conditions in Access policies or Gateway rules:

| Check | What it verifies |
|---|---|
| **OS version** | Minimum macOS/Windows/iOS/Android version |
| **Disk encryption** | FileVault/BitLocker enabled |
| **Firewall** | OS firewall enabled |
| **Screen lock** | Auto-lock timeout configured |
| **Antivirus** | Crowdstrike/Defender/etc. installed and running |
| **WARP enrollment** | Is WARP actually connected (not just installed)? |
| **Client certificate** | Is a specific certificate present in the device keychain? |
| **File exists** | Is a specific file present? (for custom agent verification) |
| **Registry key** | Windows Registry value present (custom MDM enrollment signal) |
| **Crowdstrike Zero Trust** | Crowdstrike sensor active, ZTA score above threshold |
| **SentinelOne** | SentinelOne agent active |

Posture checks run on a configurable interval and update in near-real-time. An Access policy can require: "User is in Engineering group AND device has disk encryption AND OS is up to date."

### Split Tunneling

Split tunneling defines which traffic bypasses WARP vs routes through it:

**Exclude mode (default):** All traffic goes through WARP except:
- Traffic to excluded IPs/CIDRs (e.g., local network, internal app servers)
- Traffic to excluded domains (e.g., known-good SaaS apps you trust)

**Include mode:** Only specified traffic goes through WARP:
- Explicitly listed domains (e.g., only company apps)
- Everything else goes directly to the internet

Split tunneling is configured centrally in Zero Trust → Settings → WARP Client → Device settings → Split Tunnels. Changes propagate to all enrolled devices.

### Managed Networks

WARP can detect when a device is on a trusted corporate network (office) and adjust behavior:

- **On corporate network:** Apply different Gateway policies (e.g., less restrictive, no HTTP inspection since you trust the network)
- **Off corporate network:** Apply strict policies
- **Detection mechanism:** WARP checks if a specific DNS hostname resolves to a specific IP — if it does, device is "on-network"

### MDM Deployment

Zero Trust WARP is deployed at scale via Mobile Device Management (MDM):

**Jamf (macOS/iOS):**
```xml
<!-- Jamf Configuration Profile -->
<key>organization</key>
<string>macksportreport.cloudflareaccess.com</string>
<key>auto_connect</key>
<integer>1</integer>
<key>service_mode</key>
<string>warp</string>
```

**Microsoft Intune (Windows):**
```powershell
# IntuneWin package of cloudflared installer with switches:
# /ORGANIZATION=macksportreport.cloudflareaccess.com /SERVICEMODE=warp /AUTOENROLL=1
```

After MDM push, WARP auto-enrolls without user interaction.

### WARP Connector (Site-to-Site)

WARP Connector is a server-side component that connects entire networks to Cloudflare without installing WARP on every device. Useful for:
- Branch offices (router-level connectivity)
- IoT devices that can't run WARP
- Data centers connecting to Cloudflare backbone

## Dashboard Walkthrough

**Step 1: Configure WARP for Your Organization**
1. Zero Trust → Settings → WARP Client
2. Set organization name (used for device enrollment URL)
3. Configure default device settings

**Step 2: Device Settings Profile**
1. Zero Trust → Settings → WARP Client → Device settings → Add a profile
2. Configure:
   - Allowed to leave WARP (yes/no for managed devices: no)
   - Allow device to disconnect (compliance requirement: no)
   - Gateway DNS URL
   - Split tunnel rules

**Step 3: Set Up Enrollment Rules**
1. Zero Trust → Settings → WARP Client → Device enrollment
2. Add rules: who can enroll (email domain, IdP group)
3. Only authorized users can enroll devices

**Step 4: Configure Device Posture Check**
1. Zero Trust → Settings → WARP Client → Device posture → Add new check
2. Example: Disk Encryption (macOS/Windows)
3. Enable check
4. Use in an Access policy:
   - Access → Application → Policy → Require → Device Posture → Disk Encryption

**Step 5: Monitor Enrolled Devices**
1. Zero Trust → My Team → Devices
2. See: all enrolled devices, WARP version, last seen, posture compliance status
3. Revoke a device (immediate disconnect)

## Hands-On Lab

### Prerequisites
```bash
# Download WARP client
# macOS: https://1.1.1.1/
# Windows: https://1.1.1.1/
# Or search "Cloudflare WARP" in App Store / Microsoft Store

export CF_ACCOUNT_ID="your-account-id"
export CF_API_TOKEN="your-api-token"
```

### Lab 1: Install and Enroll WARP (macOS)
```bash
# Install via Homebrew (macOS)
brew install --cask cloudflare-warp

# Check WARP status
warp-cli status

# Register with your Zero Trust organization
warp-cli registration new
# Or via the GUI: click WARP icon → Settings → Account → Log in with Cloudflare Zero Trust
# Enter: macksportreport.cloudflareaccess.com

# Connect
warp-cli connect

# Verify status
warp-cli status
# Output: Connected
```

### Lab 2: Check Split Tunnel Configuration
```bash
# List current split tunnel exclusions
warp-cli tunnel list

# Add an exclusion (traffic to this IP bypasses WARP)
warp-cli tunnel exclude 10.0.0.0/8

# Remove an exclusion
warp-cli tunnel include 10.0.0.0/8

# Check current mode (warp or doh)
warp-cli mode
```

### Lab 3: Configure Device Posture via API
```bash
# Create a disk encryption posture check
curl -X POST "https://api.cloudflare.com/client/v4/accounts/${CF_ACCOUNT_ID}/devices/posture" \
  -H "Authorization: Bearer ${CF_API_TOKEN}" \
  -H "Content-Type: application/json" \
  --data '{
    "name": "Disk Encryption Required",
    "type": "disk_encryption",
    "description": "Device must have disk encryption enabled",
    "schedule": "5m",
    "match": [
      {
        "platform": "mac"
      }
    ],
    "input": {
      "requireAll": true,
      "checkDisks": []
    }
  }'
```

### Lab 4: List Enrolled Devices
```bash
# List all enrolled devices
curl "https://api.cloudflare.com/client/v4/accounts/${CF_ACCOUNT_ID}/devices?per_page=50" \
  -H "Authorization: Bearer ${CF_API_TOKEN}"
```

### Lab 5: Revoke a Device
```bash
DEVICE_ID="your-device-id-from-lab-4"

curl -X DELETE "https://api.cloudflare.com/client/v4/accounts/${CF_ACCOUNT_ID}/devices/${DEVICE_ID}" \
  -H "Authorization: Bearer ${CF_API_TOKEN}"
# Device immediately loses access to private networks and its session tokens are invalidated
```

### Lab 6: Test Gateway Policies with WARP Connected
```bash
# With WARP connected, verify DNS is going through Gateway
# Check which DNS resolver is active
dig +short macksportreport.com
# Should resolve correctly

# Check your public IP (should show Cloudflare IP, not your real ISP IP)
curl https://ifconfig.me
# Returns a Cloudflare egress IP

# Test a Gateway policy (if you blocked social media):
curl -v https://facebook.com 2>&1 | head -20
# Should return blocked page or fail DNS

# Test a blocked malware domain:
nslookup malware-test.cf | head -5
# Should return NXDOMAIN or Cloudflare block page IP
```

## Competitive Context

| Feature | Cloudflare WARP | Zscaler Client Connector | Palo Alto GlobalProtect | Cisco AnyConnect | Tailscale |
|---|---|---|---|---|---|
| **Protocol** | WireGuard | Proprietary | IPSec/SSL | SSL/IPSec | WireGuard |
| **Always-on** | Yes (configurable) | Yes | Yes | Yes | Yes |
| **Gateway integration** | Native (CF Gateway) | Zscaler ZIA | Palo Alto NGFW | Cisco Umbrella | No SWG |
| **Device posture** | Yes (extensive) | Yes | Yes | Yes | Basic |
| **Private network access** | Yes (via Tunnel) | Yes (ZPA) | Yes (HIP) | Yes (split tunnel) | Yes (mesh) |
| **Performance** | WireGuard (fast) | Varies | Varies | Slower (SSL) | WireGuard (fast) |
| **MDM deployment** | Jamf/Intune/MDM | Yes | Yes | Yes | Yes |
| **Free tier** | Yes (personal) / Per seat (ZT) | No | No | No | Yes (< 3 users) |
| **Consumer app** | Yes (1.1.1.1) | No | No | No | No |
| **Platform support** | Win/Mac/iOS/Android/Linux | Win/Mac/iOS/Android | Win/Mac/iOS/Android | Win/Mac/iOS/Android | Win/Mac/iOS/Android/Linux |

**Key positioning:** WARP is the only enterprise device agent with a credible consumer product (1.1.1.1 + WARP) — meaning the client is battle-tested at massive consumer scale. And uniquely, WARP is part of a platform that includes DDoS protection, CDN, and full network services — not just a Zero Trust point product.
