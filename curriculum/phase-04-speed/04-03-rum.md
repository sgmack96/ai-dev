## Theory (SE-Level)

### What Is Real User Monitoring (RUM)?

Real User Monitoring captures performance data from **actual user browsers** as they navigate your site, in real time. Unlike Observatory (synthetic testing from CF edge nodes) or Origin Analytics (server-side TTFB), RUM measures the full user experience — from the moment a user navigates to your page until the page is interactive — on their actual device, on their actual network, in their actual geography.

RUM answers the question: **"What are real users actually experiencing right now?"**

### How Cloudflare RUM Works

Cloudflare RUM works through a small JavaScript beacon injected into your pages. When a user loads your page, the beacon:

1. Collects Web Vitals metrics (FCP, LCP, INP, CLS, TTFB) from the browser's Performance API
2. Records metadata: browser, OS, device type, country, network connection type
3. Sends a small JSON payload to Cloudflare's collection endpoint (non-blocking, asynchronous)
4. Cloudflare aggregates the data and surfaces it in the dashboard

The beacon is tiny (~5KB), loads asynchronously (does not block rendering), and collects data without PII.

### Privacy Model — No PII, No Cookies

Cloudflare RUM was designed privacy-first:

- **No cookies** — no persistent tracking identifiers
- **No IP addresses** stored — only country/region-level geography
- **No user IDs** — sessions are anonymous
- **No page content** — only performance timing data
- **Aggregate only** — you see percentile distributions, not individual user traces
- **GDPR-friendly** — no consent requirement under most interpretations (no personal data)

This is different from traditional RUM providers (New Relic, Datadog) that require cookie consent banners.

### Enabling RUM

Two methods:

**Method 1: Auto-injection (Cloudflare Workers / Zaraz)**
- Enable via Zaraz → Tools → Web Analytics
- Cloudflare automatically injects beacon via Zaraz on every page response
- Zero code changes to your site

**Method 2: Manual JS Snippet**
```html
<!-- Add to <head> of every page -->
<script defer src='https://static.cloudflareinsights.com/beacon.min.js'
  data-cf-beacon='{"token": "YOUR_BEACON_TOKEN"}'></script>
```

Get your beacon token from: Cloudflare Dashboard → macksportreport.com → Speed → Web Analytics → Set up → Copy Beacon Token

### Data Collected by RUM Beacon

| Metric | What It Measures | Timing |
|--------|-----------------|--------|
| **FCP** (First Contentful Paint) | First pixel rendered by browser | During page load |
| **LCP** (Largest Contentful Paint) | Largest element visible | During load |
| **INP** (Interaction to Next Paint) | Responsiveness to user input | During entire session |
| **CLS** (Cumulative Layout Shift) | Unexpected layout movement | During load |
| **TTFB** (Time to First Byte) | Network + server response time | Network layer |
| **FID** (First Input Delay) | Legacy interaction metric | First interaction |

Plus metadata:
- Country, region
- Browser (Chrome, Firefox, Safari, etc.)
- Device type (desktop, mobile, tablet)
- OS (Windows, macOS, Android, iOS)
- Connection type (4G, 3G, WiFi, broadband)
- Page URL (path only, no query strings by default)

## Deep Dive (Architect-Level)

### RUM vs Synthetic: Understanding the Gap

RUM and Observatory/synthetic monitoring measure different things. Understanding when to trust each is an architect-level skill.

| Dimension | RUM | Synthetic (Observatory) |
|-----------|-----|------------------------|
| **Data source** | Real user browsers | Headless Chromium at CF edge |
| **Network conditions** | Real — 3G, flaky WiFi, corporate proxy | Controlled — CF edge network |
| **Device conditions** | Real — 4-year-old Android, slow CPU | Standardized Chrome profile |
| **Geographic spread** | Wherever your users are | Wherever you run the test |
| **Volume** | Thousands to millions of sessions | One test at a time |
| **Actionability** | Tells you what users experience | Tells you how to fix it |
| **Best for** | Confirming a problem exists, prioritizing by user impact | Diagnosing root cause, testing fixes |

**The detective loop:** Use RUM to discover problems (p75 LCP is 4.2s for mobile users in Germany). Use Observatory to diagnose and fix them (run Observatory from Frankfurt region, see what's slow). Use RUM again to confirm the fix worked.

### Percentile Philosophy: Why p75 Matters More Than Average

Averages lie. A site with 80% of users at 1s LCP and 20% at 10s LCP has an "average" of 2.8s — which masks the fact that 1 in 5 users is having a terrible experience.

Google's Core Web Vitals threshold system uses **p75** specifically:
- Your site "passes" LCP if the 75th percentile of your RUM data is < 2.5s
- This means 75% of real users must have LCP under 2.5s to pass

Why p75 and not p50 (median)?
- p50 hides tail problems
- p99 is too harsh (outliers always exist)
- p75 is the "most users" threshold — capturing a realistic but demanding standard

For internal SLA monitoring, watch both p75 (Google's threshold) and p95 (your worst 5% of users).

### Web Vitals Attribution — Finding Root Cause in RUM

Modern RUM (and Chrome's Web Vitals library) includes attribution data:

**LCP Attribution:**
- Which element is the LCP element? (e.g., `<img id="hero">`)
- Was it a render delay or resource load delay?
- What URL is the LCP image?

**CLS Attribution:**
- Which element caused the largest shift?
- What was it before/after?
- Was it triggered by a specific event (image load, font load)?

**INP Attribution:**
- Which interaction had the worst INP? (click, keypress, etc.)
- What element was interacted with?
- How long was the processing time vs. render time?

This attribution data helps you go from "CLS is 0.3" to "The ad unit in the sidebar is causing a 0.28 CLS when it loads at 2.1 seconds post-navigation."

### Single Page Application (SPA) Support

Traditional Web Vitals are measured once per page navigation (full page load). SPAs (React, Vue, Angular apps) don't do full page loads — they do client-side route transitions.

Cloudflare RUM handles this by:
- Using the "soft navigation" spec (still evolving in Chrome)
- Re-measuring LCP after each route change
- Tracking INP across the full session (not just first navigation)

**Caveat:** SPA support is less mature than standard multi-page app support. For best accuracy with SPAs, use the `web-vitals` JS library alongside CF RUM beacon, and ensure your SPA framework is configured to fire `visibilitychange` events on route transitions.

### Filtering Dimensions in RUM

The real power of RUM is segmentation. You can filter by:

- **Country** — Find that your site is fast in the US but slow in Brazil (→ geography of your origin)
- **Device type** — Find that mobile users have 3x worse LCP than desktop (→ responsive image issue)
- **Browser** — Find that Safari has worse INP than Chrome (→ browser-specific JS issue)
- **Connection type** — Find that 3G users have terrible FCP (→ asset size issue)
- **Page path** — Find that `/pricing` has much higher CLS than other pages (→ specific component)

This segmentation is what separates "our performance is 2.5s LCP" from "our performance is 2.5s LCP on desktop in North America, 4.8s LCP on mobile in Southeast Asia" — actionable vs. vague.

### Using RUM to Justify Optimization Investment

RUM creates the business case. Workflow:

1. **Quantify the problem:** "23% of mobile sessions have LCP > 4 seconds"
2. **Map to user impact:** "You have 50,000 mobile sessions/month → 11,500 sessions with poor LCP"
3. **Benchmark against competition:** "Your competitor scores p75 LCP of 1.8s on mobile"
4. **Tie to revenue:** Cite Google's research: 1 second LCP improvement → ~7% conversion increase
5. **Project the fix:** "Enabling Polish + Early Hints typically reduces LCP by 800ms–1.5s"
6. **Show the delta in RUM** after the fix ships

## Dashboard Walkthrough

### Step 1: Navigate to RUM
```
macksportreport.com → Speed → Real User Monitoring
```
*(Note: May appear as "Web Analytics" in some dashboard versions)*

### Step 2: Enable RUM if Not Yet Active
- Click "Set Up" → Copy beacon token
- Add JS snippet to `<head>` or enable via Zaraz
- Data appears within ~15 minutes of first real user visit

### Step 3: Read the Overview Panel
- **Core Web Vitals summary:** LCP, INP, CLS with p75 values and pass/fail status
- **TTFB card:** p75 from real user perspective
- **Session count:** Total sessions collected in time window
- **Traffic trend:** Sessions over time

### Step 4: Drill into LCP
- Click "LCP" card
- See histogram of LCP values across all sessions
- Check p50, p75, p95 tabs
- Filter by: Country, Device type, Browser
- Look for segments with dramatically different LCP

### Step 5: Analyze by Geography
- Click "Countries" breakdown
- Sort by p75 LCP descending
- Identify outlier countries with poor performance
- For worst-performing countries: consider regional origin deployment, Argo, or Tiered Caching

### Step 6: Mobile vs Desktop Comparison
- Filter: Device = Mobile → record p75 LCP, CLS, INP
- Filter: Device = Desktop → record p75 LCP, CLS, INP
- Compute ratio — if mobile > 2x desktop, image optimization issue or render-blocking JS is the culprit

### Step 7: Page-Level Analysis
- Filter by specific URL paths
- Identify which pages have the worst Core Web Vitals
- Prioritize: high traffic + poor score = biggest opportunity

## Hands-On Lab

### Prerequisites
- macksportreport.com on Cloudflare
- RUM beacon installed (use snippet method for lab)
- At least some real traffic (or simulate with curl/browser visits)

### Lab 1: Install RUM Beacon

```bash
# Step 1: Get your beacon token from CF API
curl -s -X GET "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/web3/hostnames" \
  -H "Authorization: Bearer $CF_API_TOKEN" | jq '.'

# Step 2: Create/get Web Analytics site
curl -s -X POST "https://api.cloudflare.com/client/v4/accounts/$ACCOUNT_ID/rum/site_info" \
  -H "Authorization: Bearer $CF_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "auto_install": true,
    "host": "macksportreport.com",
    "zone_tag": "'$ZONE_ID'"
  }' | jq '{token: .result.lite_id, snippet: .result.snippet}'
```

### Lab 2: Verify Beacon is Firing

```bash
# Open your site and check network tab for beacon request
# Or simulate it with curl:

BEACON_TOKEN="your_beacon_token_here"

# Manually send a test beacon payload (mimics what a browser sends)

    },

echo "Beacon payload sent. Check RUM dashboard in ~60 seconds."
```

### Lab 3: Pull RUM Data via GraphQL API

```bash
# Query RUM data using CF GraphQL Analytics API
curl -s -X POST "https://api.cloudflare.com/client/v4/graphql" \
  -H "Authorization: Bearer $CF_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "
      query {
        viewer {
          accounts(filter: { accountTag: \"'$ACCOUNT_ID'\" }) {
            rumWebsiteMetricsAdaptiveGroups(
              filter: {
                date_gt: \"2024-01-01\",
                date_lt: \"2024-01-08\",
                siteTag: \"'$BEACON_TOKEN'\"
              }
              limit: 100
              orderBy: [count_DESC]
            ) {
              count
              dimensions {
                deviceType
                countryName
                browserFamily
              }
              quantiles {
                largestContentfulPaintP75
                cumulativeLayoutShiftP75
                firstInputDelayP75
                timeToFirstByteP75
              }
            }
          }
        }
      }
    "
  }' | jq '.data.viewer.accounts[0].rumWebsiteMetricsAdaptiveGroups[] | {
    sessions: .count,
    device: .dimensions.deviceType,
    country: .dimensions.countryName,
    lcp_p75_ms: .quantiles.largestContentfulPaintP75,
    cls_p75: .quantiles.cumulativeLayoutShiftP75,
    ttfb_p75_ms: .quantiles.timeToFirstByteP75
  }'
```

### Lab 4: Build a RUM Performance Report

```bash
#!/bin/bash
# rum-report.sh — Generate a summary RUM performance report

CF_API_TOKEN="${CF_API_TOKEN}"
ACCOUNT_ID="${ACCOUNT_ID}"
BEACON_TOKEN="${BEACON_TOKEN}"

START_DATE="${1:-$(date -u -v-7d +"%Y-%m-%d" 2>/dev/null || date -u --date="7 days ago" +"%Y-%m-%d")}"
END_DATE="${2:-$(date -u +"%Y-%m-%d")}"

echo "=================================="
echo "  RUM Performance Report"
echo "  macksportreport.com"
echo "  $START_DATE → $END_DATE"
echo "=================================="
echo ""

RESULT=$(curl -s -X POST "https://api.cloudflare.com/client/v4/graphql" \
  -H "Authorization: Bearer $CF_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "query { viewer { accounts(filter: { accountTag: \"'$ACCOUNT_ID'\" }) { rumWebsiteMetricsAdaptiveGroups(filter: { date_gt: \"'$START_DATE'\", date_lt: \"'$END_DATE'\", siteTag: \"'$BEACON_TOKEN'\" }, limit: 1000, orderBy: [count_DESC]) { count quantiles { largestContentfulPaintP75 cumulativeLayoutShiftP75 firstInputDelayP75 timeToFirstByteP75 } } } } }"
  }')

echo "=== Overall Core Web Vitals (p75) ==="
echo "$RESULT" | jq '.data.viewer.accounts[0].rumWebsiteMetricsAdaptiveGroups | {
  "Total Sessions": (map(.count) | add),
  "LCP p75 (ms)": (map(.quantiles.largestContentfulPaintP75 * .count) | add) / (map(.count) | add),
  "CLS p75": (map(.quantiles.cumulativeLayoutShiftP75 * .count) | add) / (map(.count) | add),
  "TTFB p75 (ms)": (map(.quantiles.timeToFirstByteP75 * .count) | add) / (map(.count) | add)
}'

echo ""
echo "=== Pass/Fail Assessment ==="
LCP=$(echo "$RESULT" | jq '(.data.viewer.accounts[0].rumWebsiteMetricsAdaptiveGroups | (map(.quantiles.largestContentfulPaintP75 * .count) | add) / (map(.count) | add))')

if (( $(echo "$LCP < 2500" | bc -l 2>/dev/null || echo 0) )); then
  echo "✅ LCP: PASSING ($LCP ms < 2500ms)"
else
  echo "❌ LCP: FAILING ($LCP ms > 2500ms)"
fi
```

## Competitive Context

| Dimension | Cloudflare RUM | Google CrUX | New Relic Browser | Datadog RUM |
|-----------|---------------|-------------|-------------------|-------------|
| **Data source** | CF beacon (JS) | Chrome browser telemetry | JS agent | JS agent |
| **Privacy** | No PII, no cookies | Anonymized, aggregate | Cookie-based, PII risk | Cookie-based, PII risk |
| **Consent required** | No (no PII) | No (passive Chrome data) | Often yes (GDPR) | Often yes (GDPR) |
| **Setup** | Single JS snippet | No setup (passive) | Agent install + config | Agent install + config |
| **Data freshness** | Near real-time | 28-day rolling average | Near real-time | Near real-time |
| **Filtering dims** | Country, device, browser, path | Limited | Full session replay | Full session replay |
| **Session replay** | No | No | Yes (paid) | Yes (paid) |
| **Root cause tracing** | No | No | Yes (distributed tracing) | Yes (distributed tracing) |
| **Cost** | Included in CF plans | Free | $$$ | $$$$ |

**SE Positioning:** CF RUM hits the sweet spot — zero setup, privacy-safe, real data, free with your CF subscription. It won't replace Datadog for deep debugging, but for the "are my users experiencing my site well?" question, it's the fastest answer available.
